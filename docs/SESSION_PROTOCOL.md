# Paco — Session Protocol

> How we work together. The methodology developed across the project's first few weeks.
> A future-Claude reading this should behave like the Claude that produced the working v0.3.x.

## My role

I'm the architect/planner. I don't write code directly. I help Omar:

- **Plan milestones** before code gets written. Scope, sequence, edge cases, decisions.
- **Write prompts to send to Claude Code** (in his terminal). Claude Code is the implementer.
- **Review Claude Code's plans and diffs** before they ship. Catch scope creep, security holes, missed edge cases.
- **Push back on his ideas when warranted** and steelman his ideas when they're sharper than mine.
- **Maintain the four living docs** (`docs/PROJECT_STATE.md`, `docs/roadmap.md`, `docs/backlog.md`, `docs/SESSION_PROTOCOL.md`) — all repo-canonical as of v0.5.4, maintained in place by Claude Code as part of each milestone's close commit.

Omar is in the loop for every meaningful decision. He sits between me and Claude Code, copy-pasting prompts and diffs. The friction is the value — it's where his product judgment lives.

## How a typical session goes

1. **Open the session** with the current state. He'll often say "ready for M3.2" — I look at the roadmap, propose scope, ask any decisions before drafting the prompt.
2. **I draft a prompt for Claude Code.** Includes context (read the four living docs), explicit scope, constraints, and a "stop after plan, wait for approval" instruction.
3. **He sends to Claude Code, brings back the plan.** I review against the original spec. Push back if anything's off — security, scope, design coherence, missed edge cases.
4. **He approves the plan with Claude Code.** Diffs roll in. He approves each one (manual mode, never auto-approve git/file operations).
5. **He smoke tests after the commits land.** If something breaks, he tells me, we diagnose and send a follow-up prompt. If clean, we tag.
6. **Tag the milestone** with `git tag -a v0.x.y -m "..."`. Single source of truth for shipped work.

## Operating principles

### Plan before code

Every milestone starts with a plan from Claude Code. Approval is explicit, not implicit. Plans surface decisions Omar should weigh in on. We never "just start coding" — even small tasks benefit from a plan that names the files and decisions.

### Scope discipline

A milestone fits in 1-2 sessions ideally. If it's bigger, split it. Saying "M2 has 4 sub-milestones" is better than "M2 is one thing that takes 4 sessions." The split:
- Forces clearer thinking about dependencies
- Lets Omar tag intermediate progress
- Makes the work feel more shippable

### Manual approval, every time

Claude Code asks "do you want to proceed?" before bash commands, file writes, and git operations. Omar always picks "1. Yes" — never "Yes and don't ask again." The 1-2 second cost of approval is the last guard against destructive accidents (force-push, drop tables, leak secrets to git). Speed is not worth removing this layer.

### Push back when warranted

Don't be sycophantic. If Omar proposes something I think is wrong, I say so directly, give my reasoning, then offer a counter-proposal. He's been right when he pushed back on me ("the architect/Claude Code multi-agent vision is more nuanced than your strawman"). I should do the same in reverse.

### Be honest about session limits

When the conversation gets long, suggest closing. When his weekly limits are near, say so. The product is better served by a fresh chat than a degraded one.

## Conventions for prompting Claude Code

When drafting a prompt, I always include:

- **"Read the four living docs first (`docs/PROJECT_STATE.md`, `docs/roadmap.md`, `docs/backlog.md`, `docs/SESSION_PROTOCOL.md`). Then plan mode."** — primes Claude Code with project context, prevents it from drifting. (`CLAUDE.md` also points at all four at task start.)
- **Explicit scope** — what's in, what's out. No ambiguity.
- **Constraints** — TypeScript strict, no `any`, lint at end, single commit, etc.
- **Edge cases** — known failure modes called out explicitly.
- **"Stop after plan. Wait for approval."** — the most important sentence in every prompt.

When the prompt has user-provided content (like idiom JSON or boilerplate text), I include it verbatim in the prompt so Claude Code doesn't paraphrase or "improve" it.

When a milestone touches sensitive infrastructure (auth, schema migrations, admin keys), I add explicit security checkpoints. E.g., "the admin client must use `import 'server-only'`, not just a comment."

## Things I've learned to watch for

- **Comments instead of guards.** "Server-only by convention (comment-guarded)" is not actually server-only. Real `import 'server-only'` is the only real guard.
- **Auto-flips.** Loading-state UX should never auto-flip when complete. The user clicks to advance. Respects their attention.
- **State machine completeness.** Plans for stateful UI should explicitly enumerate every transition, including error → retry, navigation → cancel, and post-success cleanup.
- **Cache before API.** When checking duplicate-or-new, query the database first, only call Anthropic if needed. We made this mistake in M3.1 and had to fix it as a follow-up.
- **Image weights.** Static assets in `public/` add up. Compress aggressively (Squoosh quality 65, 1200px max width) — we hit 31MB then trimmed to 7.5MB.
- **iOS Safari quirks.** Inputs under 16px trigger zoom-and-stay. Lock viewport scale + use 16px minimum.
- **Plan compaction.** Long Claude Code sessions get auto-compacted, losing context. Start fresh sessions for new milestones to avoid drift.
- **Diagnosis is hypothesis until checked.** A confident, code-cited root cause can survive several review rounds and still be false. When a diagnostic exists (a SQL query, a log, a feature flag), run it before building apparatus on top of the diagnosis. M5.0b's orphan misdiagnosis cost multiple rounds of architecture built for a non-existent condition; the one-line `SELECT ... LEFT JOIN review_cards ... WHERE rc.id IS NULL` that killed it could have been run in round one.
- **Send Claude Code the actual mockup images, not prose translations.** When designs exist, attach them. Prose-translated specs lose visual intent; Claude Code builds a plausible-looking thing that drifts from the design, and the drift only surfaces on smoke-test screenshots. M5.0b's tan preview card, cramped button, and missing divider were all drift from prose specs; once mockups were attached directly, the next pass landed. (M5.0c generated the mockups via Claude Design first, then attached them — no drift.)
- **A computed-style HTML export is not a usable mockup, and visual polish needs the cropped component — not the whole board.** M5.2's nav took ~5 feedback rounds (serif-vs-sans, colored-border-vs-shadow, pill height, logo placement) because Claude Code got a 4MB computed-style dump (no class names) and rebuilt from token-guesses. It converged only once a cropped image of the exact element (the pill row) was attached and we iterated on screenshots. For fine visual matching: attach a clean crop of the specific component and expect screenshot-driven iteration — a style-dump export is worse than a PNG.
- **Pattern-match cumulative "it's handled" claims with extra suspicion.** When prior sweeping summaries ("comment-only change" / "this is the root cause" / "everything else is already done from prior commits") have been wrong, the next one needs to demonstrate per-item, not assert. Same bar applied evenly — not unkindness, the track record requires proof.
- **Bundled AskUserQuestion options can smuggle a sub-decision past you.** When Claude Code says a choice was "confirmed with the user," check whether it was a standalone pick or one option that bundled several decisions together. M5.0c's "keep a Home preview" + "remove the pinned action bar" pair rode in on a single bundled selection Omar made for other reasons; demanding per-item provenance is what surfaced it.
- **Post-tag polish drifts the tag from shipped reality unless the tag moves.** When a fix or polish pass lands after a milestone is tagged, move the tag to the final commit — the tag is the single source of truth for shipped work, and `PROJECT_STATE.md` is keyed to it. Claude Code may call leaving the tag behind "consistent with how we do it" when it is the opposite (M5.0b moved its tag to the final fix commit). M5.0c shipped a cleanup commit plus a row redesign past v0.5.2; the tag was re-pointed to the final commit to restore tag = shipped.
- **A UNIQUE constraint on an FK column flips PostgREST embed cardinality.** Adding a `UNIQUE` (or unique index) on the foreign-key column of a child table changes how PostgREST infers the relationship: a `to-many` embed (returns an **array**) becomes `to-one` (returns a **single object**, or `null`). This silently breaks every embed read that indexes the array — `data.child[0]`, `cards?.[0]`, `as Child[]` — because the value is no longer an array; the query still succeeds and rows still return, so it looks like a cosmetic/null-data bug, not a schema change. (M5.1: adding `UNIQUE(review_cards.word_id)` made `words → review_cards` to-one; the whole `/words` list silently showed every word as "Nouveau / 0 révision". Data was intact — a read-shape bug.) **Rule: when adding such a constraint, audit _every_ embed read of that relationship** (`grep` the table name in `.select(...)` embeds and every `[0]` / array-typed read) and normalize the read to tolerate both shapes — e.g. `oneEmbed()` in `lib/word-status.ts`:

  ```ts
  export function oneEmbed<T>(embed: T | T[] | null | undefined): T | null {
    if (Array.isArray(embed)) return embed[0] ?? null
    return embed ?? null
  }
  ```

  Normalizing (rather than dropping the constraint) is the correct end state when the to-one cardinality is a true invariant: PostgREST is now reading the relationship correctly, and the old array-indexing relied on the prior to-many mis-inference. The completeness grep also surfaced a 4th consumer (the `/add` duplicate-hit path) that recall had missed — **grep, don't recall.**
- **Repo docs can silently diverge from the planning docs.** Historically `PROJECT_STATE` / `SESSION_PROTOCOL` lived only in the Claude Project while `roadmap` / `backlog` were mirrored and drifted. A doc reconciliation built on "the repo matches the Project" was caught by a verify gate (do the markers exist on disk?) before any edits. **Verify on-disk state before reconciling — the repo is ground truth for the repo.** (Retired as of v0.5.4 — see the repo-canonical note below — but the verify-first habit stays.)
- **Living docs are repo-canonical (as of v0.5.4).** All four live in `docs/` and are maintained in place by Claude Code as part of each milestone's close commit (targeted edits, not rewrites). The Project no longer holds doc copies; the architect is given the current docs at session start. This retires the delete-and-reupload loop and the repo/Project divergence (cf. the M5.1 fold that sat unapplied until the v0.5.4 docs pass).
- **One sampled cell hides a systemic bug; verify the full paradigm.** v0.6.4's audit table first showed only *tú-present* for the stem-changers and *usted-imperative* for the irregulars. The real defect was an imperative builder that bypassed the irregular table for **all 16** fully-irregular verbs — invisible if you only check the cell you sampled. The fix-admission rule became "a verb is trusted only if its **full** paradigm matches a reference," and that full check is exactly what *caught* the blind spots that got excluded (poder's `pudiendo`, creer's `creíste`). Same shape as the "demonstrate per-item, not assert" lesson: a guard authored from the production code's own output (or from one verified cell) inherits the production code's blind spots — author the high-risk cells from an **independent reference** (a clean-room reimplementation for the regular class; hand-verified golden forms for the irregular class), so the test verifies rather than echoes.
- **Lexical validation can't catch grammatical malformation; gate on the structured flag and degrade safe.** v0.6.5's `"ti acuestas"` passed the per-token spellcheck because every token is a real Spanish word — the error was the *wrong proclitic for the person*, which no dictionary lookup sees. The fix keyed off the enrichment's *reflexive flag + person* (the structured signal), not the surface text, and split detection (a token outside `{me,te,se,nos,os}` is definitively wrong — no person needed) from correction (the person picks the replacement), so annotation drift degrades to "store the lemma," never "store the typo." General shape: when a check needs grammatical/structural knowledge the lexical layer lacks, gate on the upstream structured signal and choose a fallback that fails toward the safe artifact. (Reinforced the inline→tested-helper rule too: both v0.6.5 fixes were lifted into pure unit-tested helpers — `resultHintExample`, `correctProcliticReflexive` — never inlined at the call site, per the `chooseQcmCue` regression.)
- **Don't pixel-chase alignment to a component that's being replaced, and trust "only this screen looks odd."** The Découvrir grid burned ~10 rounds trying to align tile edges to bottom-nav items — two independent layouts that don't share a structure, against a nav that M5.2 deletes. The user's repeated "only this screen looks odd, the others are fine" was the correct signal; re-arguing "it should be identical" prolonged it. When the thing being aligned-to is slated for a rebuild, defer the alignment to that rebuild.
- **Enforce an allowlist structurally, not by convention (M5.5a).** The brief asked that Fraunces be "exposed through ONE narrow mechanism — the only path by which it is ever applied, so the allowlist is enforced structurally, not by convention." The shipped guard: load the font as a raw `<html>` CSS variable but **deliberately do NOT register it in the Tailwind `@theme`**, so no `font-fraunces` utility is ever generated — there is no class to sprinkle. The single component that names `var(--font-fraunces)` takes a **required closed-union prop** (`kind: 'listo' | 'casi' | 'count'`), so TypeScript rejects any off-allowlist use at the call site. Same family as the "comments instead of guards" lesson: a documented rule ("only use Fraunces on X") is convention; an absent utility + a closed union is a structural guard. When asked for an allowlist/restriction, prefer making the wrong thing *impossible to express* over documenting that it's disallowed.
- **A "paint" pass can be mostly already-done — say so, don't manufacture diffs (M5.5a).** Board §05 named three nav changes (active = amber fill, labels = Lora bold, brand lockup kept); the M5.2 nav already satisfied all three, so the honest diff was tiny (route active through the shared selection treatment → it gains the §04 short shadow + ivory text). The right move was to state "already compliant" per item and change only the genuine delta, **not** to repaint the whole nav to look busy — and to explicitly *not* touch the inactive accent-tinted border (a deliberate M5.2 choice, outside the three named changes). Scope discipline cuts both ways: don't under-build, but don't invent changes to look thorough.
- **A "presentation-only" re-skin earns the label by where the new logic lives — a pure projection over the existing model, never a fork of it (M5.5b).** The Words cluster had to recolor/merge status pills and add a 4-dot gauge *without* touching the locked 6-band `lib/word-status.ts` taxonomy. The discipline that made "presentation-only" true and checkable: the gauge is `getMasteryGauge = getFamiliarity rendered as quarters` (one pure function over the existing band predicate, unit-tested, image `{0,1,2,4}`), and the pill merge is a **render-layer label→kind map** consuming `getWordStatus().label` — neither re-derives bands, re-reads stability thresholds, or branches on FSRS state. The tell that you've drifted from "projection" into "second taxonomy": the new code starts comparing `stability` to `7`/`30` itself, or re-checks `State.Relearning`. If a merge needs two source bands to render one label (En cours + En apprentissage), keep both bands intact underneath and let a *separate* axis (the gauge) carry the distinction — don't collapse the model to match the label.
- **A presentation-only refinement can change the projection's thresholds without touching the model — the line is whether the *bands* move (M5.5b).** The gauge first shipped `{0,1,2,4}` (the 3rd dot never filled — a deliberate "steps at Mémorisé" reading); the follow-up wanted all four dots to fill smoothly. The fix added a 15-day split *inside* the existing 7–30 band — a finer **display** bucketing that's still a pure function of `stability` — without adding a status band, moving the 7/30 partition edges, or changing `getWordStatus`/`isMemorized`. The test that the change is still "presentation-only": the status partition and its predicates are byte-identical; only the gauge function's internal cut points moved. A projection is free to be as coarse or fine as the display wants, *as long as it reads the model rather than redefining it.*
- **Two screens showing the same field two ways → one shared formatter both call, not parallel per-screen formatting (M5.5b).** The /words detail and the /add card both rendered the grammatical POS, divergently (and an interim pass reused discovery's spelled-out `posEyebrow` on the detail, which *dropped gender* — `n.f.` → bare "NOM" — exactly the info Spanish learners need). The board review's fix: pick the representation that carries the most info (inline abbreviated — gender rides in `n.m.`/`n.f.`) and put it in **one** shared `posAbbrev(pos)` that *both* cards route through, so they can't drift; leave `posEyebrow` to its remaining single caller (the discovery deck) rather than overloading it. The tell you're about to fork: you're about to format the same field at a second call site. Reuse means one helper owns the rule for every caller; a per-screen format (or stretching a helper meant for a different screen) is a second copy wearing a reuse label. Lock the shared map with a test so a later edit can't silently desync the two screens.
- **Verify a mock's data assumptions against the schema before reproducing them; mocks carry placeholders (M5.5b).** The approved detail mock drew a phonetics line (`/me aˈkweɾ.ðo/`) and labeled the confusables chips "Mots de la même famille." Both were mock fictions: there is no IPA field in the enrichment schema, and the `distractors` field is MCQ confusables, not family words. Reproducing them faithfully would have shipped an empty/fake phonetics line and a mislabel that miseducates. **Grep the schema/enrichment shape for the field before building UI that assumes it exists** — when it doesn't, omit (the mock was placeholder) or keep the honest label, and record *why* in the close so the divergence reads as deliberate, not missed.
- **When a foundation primitive and a later high-fidelity screen disagree, reconcile the primitive to what ships — don't leave it orphaned (M5.5b).** M5.5a defined `SELECTION_PERSISTENT` as "crème+ tint + 1.5px amber border"; the board's actual À réviser row uses a softer `tinted-border` + an amber leading dot (amber border on every due row is too loud). Rather than render the row board-exact and let the constant rot, soften the constant to the board treatment and have the row consume it — so the single source of truth matches reality and the next cluster inherits the right style. State which variant is canonical (here: there is no amber-border variant in use) so a future reader doesn't re-add the loud one.
- **`setPointerCapture` on a link/button's ancestor silently kills desktop click-through.** M5.4b's `SwipeRow` called `setPointerCapture` on `pointerdown` on the `<div>` wrapping the row's `<a>`; on desktop mouse this retargets the synthesized `click` to the capturing div (the anchor's *ancestor*), so the click never reaches the `<a>` and navigation fails — most clicks did nothing, intermittently one got through. Touch was unaffected (touch-derived clicks hit-test to the anchor regardless of capture), which is the diagnostic tell: *mouse broken, touch fine* points at pointer capture, not the click-guard flag. **Fix:** acquire capture LAZILY, only after the drag crosses the tap-slop — never on `pointerdown` — so a clean click is byte-for-byte the plain-`<Link>` path. Prefer this over an always-`preventDefault`+programmatic-navigate guard: the latter "works" but silently kills cmd/ctrl/middle-click open-in-new-tab. (Also: the diagnosis was reached by static trace + known Chromium semantics, not a live browser repro — but the fix was chosen to be robust to the exact sub-mechanism, since a clean click now shares the unchanged Link path either way.)
- **A "re-skin of shipped behavior" brief can still hide genuinely new surface — diff the boards against the actual code before assuming pure CSS (M5.5c).** The Add-flow brief said "nothing here is new… a re-skin across all six states." But reading the live code against the six boards showed two boards (④ duplicate, ⑥ success) had **no current screen equivalent** — the *behaviors* shipped (duplicate detection routed to the full fiche; single save was a button-flip; bulk similaires was a background toast), but the *screens the boards draw* did not exist, and building them needed real decisions (drop the duplicate's inline reset-schedule? await the background bulk-add?) plus additive data plumbing (`save` must return the id; `enrich` must return card `state`/`stability`) for the shared `WordRow` to render. The brief's keep-list phrase "the WordRow reuse on the duplicate + success states" *presupposed* a reuse that wasn't there yet — it meant "reuse WordRow when you build them," not "it's already wired." Lesson: when a brief frames work as a pure re-skin, **enumerate each target screen against the running code and surface the deltas per-item before planning** (asked Omar about ④ and ⑥ before building) — don't let "re-skin" smuggle new screens + backend changes in under a CSS-only mental model. Same family as the per-item-provenance and demonstrate-don't-assert lessons.
- **A CSS one-shot animation on a persistent React node fires once for the whole screen, not once per state change — force a remount to re-fire it (M5.5c).** The loading checklist's per-phase "ring pop + check draw-on" never replayed as each phase resolved: the indicator `<span>` sat at a fixed position in the row, so React reused the same DOM node and just patched its className/children — the one-shot `animation` had already run on first mount and never restarts on a class swap. Fix: **key the indicator element by its `state`** (`key="done"|"active"|"pending"`), so a `pending→active→done` flip changes the key → React unmounts/remounts the node → the one-shot plays fresh each phase. General rule: a CSS one-shot (`animation … both`, no `infinite`) only plays on mount; to re-trigger it on a state change either remount via a changing `key`, toggle the class off-then-on across a frame, or restart on `animationend`. (Infinite pulses don't have this problem — they're always running.)
- **A bottom CTA that clips on every device is a wrong fixed-height assumption, not a safe-area bug — diff it against a screen whose CTA works (M5.5c).** The ¡Listo! "Voir la fiche" button was cut mid-height on mobile AND desktop. The mobile-AND-desktop symmetry was the tell: a safe-area/notch issue would be mobile-only. Root cause: that one screen bottom-anchored its CTA inside a hard-coded `min-h-[calc(100svh-4rem)]` column + `flex-1` spacer, and the `4rem` underestimated the real two-row `TopNav`, so the column ran taller than the viewport and pushed the CTA below the fold. The fix wasn't to tweak the calc (the next nav change would re-break it) — it was to **reuse the pattern from the screen that already works**: the fiche pins its bar with the shared `StickyActions` (`position:fixed; bottom-0` + `env(safe-area-inset-bottom)`), height-independent. When one screen's bottom element misbehaves and a sibling's doesn't, diff their layout containers and adopt the working one rather than patching the broken arithmetic. (Same family as "trust 'only this screen looks odd'.")
- **When a second screen needs the same polished motion, extract the choreography into one shared component — don't re-implement it (M5.5d).** Discovery's generation loader is "the same Paco-creuse wait" as `/add` (its handoff CSS was literally byte-identical to `/add`'s, only a class prefix differed). Rather than copy the timeline + phase-rows + floor/reveal gate into `DiscoverClient`, the shared `LoadingChecklist` was extracted from `/add`'s `LoadingIdiom` and **both** consume it (parametrized by title/phases/ready/onReveal + pacing knobs); the keyframes already lived in `globals.css` so nothing was duplicated there either. `LoadingIdiom` kept only what's genuinely add-specific (¡Listo!, error, the CTA). The tell you should extract rather than copy: you're about to paste an effect/timeline you already wrote elsewhere, changing only labels/constants — those differences are *props*, not a reason for a second copy. (Same family as the shared-`posAbbrev` and shared-`WordRow` reuse calls.) Verify the extraction didn't regress the original screen (re-tested the `/add` loading flow after the refactor).
- **When the brief enumerates a component's contents, the brief wins over an extra flourish in the board (M5.5d).** The Discovery board drew a speaker button on the swipe card; the brief enumerated the card as exactly "word + POS + gloss + one example." Adding the speaker would also have put an interactive child on a `setPointerCapture` swipe surface — the M5.4b click-eating trap. Omitted it, matching the brief, and recorded *why* (brief enumeration + no audio URL on the compact deck card + the capture risk) so the board divergence reads as deliberate. When the prose brief and the visual board disagree on a small inclusion, the brief's explicit enumeration is the controlling spec — flag the divergence, don't silently reproduce the board.
- **Keep the verdict (display) layer and the rating (scheduling-input) layer separable — you can re-skin one without touching the other (M5.5e).** The board moved the "right verb / wrong form" case from a ¡Uy! face to a neutral ¡Casi! face. The naive read is "it's now an 'almost' → also bump its suggested rating to Difficile" — but that changes what's written to FSRS. Omar's call was to change ONLY the verdict layer (`QUALITY_TO_VERDICT.wrongForm: 'wrong'→'close'`) and leave `computeRating` byte-identical, accepting a minor verdict↔suggested-pill mismatch over a scheduling-input shift. The clean separation that made this trivial: `computeRating` (the FSRS rating) and `QUALITY_TO_VERDICT` (the face/colour) already consume the same `BlankQuality` independently, so the display could move while the rating stayed put. When a re-skin brushes against grading/scheduling, find the seam between "what it looks like" and "what it writes," and confirm which layer the change is actually allowed to touch (ask if the brief's parenthetical mapping diverges from the shipped code — here "keep the mapping" meant *don't touch computeRating*, not *make it match the board's summary*).
- **A motion addition needs its honest still-state designed up front, gated structurally (M5.5e).** The auto-advance countdown is genuinely new behavior, not a re-skin. The non-negotiables that came with it: the bar is a NEUTRAL clock (faint), never a verdict colour or amber, so it can't be misread as a signal; the JS state machine (counting/paused/stopped) is the completion authority and is independent of the CSS fill, so pause/override/↵ all work even if the animation is off; and the `pacoCount` fill lives ONLY in the `prefers-reduced-motion: no-preference` block while a per-second numeric (driven by the same interval) carries the reduced-motion case — the countdown advances identically either way. Same family as the loading-choreography rule: motion is additive, the state it conveys must read without it. Also log the a11y escape hatch (a "disable auto-advance" setting) the moment you add auto-motion to a core loop.
- **When two design sources disagree on the same detail, the prose brief wins over the canvas README — and say so (M5.5f).** The brief's item-2/item-4 specified the verb Indice tiers as T1=form / T2=conjugation-table, while the README §③ prose still said verb T1=table (stale from an earlier iteration). Built per the brief and recorded the divergence in the close, so a future reader doesn't "fix" it back to the README. The brief is the controlling instruction; the canvas illustrates. (Same family as "the prose brief wins over an extra board flourish" from M5.5d — keep a running habit of naming which source you followed when they conflict.)
- **Replacing a penalty with a cap: delete the old penalty in the same change, and prove no double-count with a test (M5.5f).** The tiered hint cap superseded the single `hintUsed` −1 penalty in `computeRating`. The risk is leaving both in (the answer gets docked twice). The discipline: change the *signature* (`hintUsed: boolean` → `hintLevel: number`) so every call site must be revisited (the compiler finds them), delete the old `if (hintUsed) rating -= 1` line in the same edit, and add explicit cap tests (0→4/1→3/2→2/3→1) plus a "low base isn't raised" and a "MCQ unaffected" case. Also confirm where the old penalty actually lived before generalizing — here it was écriture-only (the mc branch never applied it), so the cap stays écriture-only rather than silently introducing a new MCQ penalty. A signature change is a feature, not a chore: it turns "did I update every caller?" into a compile error.
- **Normalize possibly-malformed shapes at the data boundary, not at each render site (M5.5e).** A new écriture context chip rendered `definition.fr` raw and crashed at card 10 on a word whose `definition.fr` was a nested `{es,fr}` object (the historic pre-M2.5 weak shape that survived in a few rows). The wrong fix is a guard at the one render that crashed — the same malformed field is read by the MCQ "Voir en français," the ¡Uy! reveal, and the session recap, so a per-site guard leaves the others latent. The right fix: a tested `flatBilingual(value, side)` coercion applied once in a **single shared row→`ReviewCard` mapper** (`mapReviewRow`, used by both the server page and the client "Encore N" refetch), so every consumer downstream is guaranteed plain strings. When a render crashes on a data shape, fix it where the data enters the typed model (and make the type honest), not where it happens to be rendered first — and centralize the mapping so the two fetch paths can't normalize differently. (When a `{es,fr}`-style object hits React with "Objects are not valid as a React child," the offending field is being read one level too shallow — find the field, render the side, normalize at the source.)
- **A CSS-keyframe `width` fill can render static in practice; a JS-driven width is the robust path for a progress bar you must SEE move (M5.5e).** The countdown bar used `@keyframes pacoCount{from{width:0}to{width:100%}}` and never visibly filled on device (while the completion timer still fired). Rather than chase why the keyframe didn't paint, the fix drove the width from the existing countdown interval (`progress = 1 − remaining/total` → inline `width` + a short `width` transition) — the bar is now guaranteed to track real elapsed time, and the same interval stays the completion authority. Keep the honest reduced-motion branch a separate render (a numeric readout), not the same animated element with motion disabled.
- **Match the success message to whether the work actually completed — confirmed vs in-progress (M5.5c).** Board ⑥ drew both success variants with definitive past-tense copy ("N mots ajoutés"). But the single-word save is **awaited** (genuinely done → "« mot » est dans ton deck" is honest), while the multi/similaires add is **fire-and-forget background** (~20 s) — so claiming "ajoutés" would assert a success that hasn't happened. Omar's call: keep the single confirmed, reframe the multi as in-progress ("N mots en route … tu peux continuer à naviguer"). The honest framing has a cost (a failed background add isn't surfaced inline) — so it was **logged as a pre-soft-launch backlog item**, not swept under. Lesson: a design mock's "done" copy is an assumption about timing; verify against whether the mutation is awaited, and where it isn't, say "in progress" and log the dropped error-surfacing rather than over-claiming.
- **A "shown-once" flag must flip when the thing is actually shown — not when you decide it's owed (M5.5g).** Relocating the dictionary-unlock celebration to review-end, the obvious wiring was "evaluate the crossing at the bilan, flip `dictionary_unlocked`, then show the takeover." But flipping at *evaluate* time means a user who reaches the bilan and quits before tapping an exit has a true flag with no celebration ever shown — and the Home safety-net (which fires only while the flag is false) is now suppressed, so the reveal is lost permanently. The fix: split the action into a **read-only `evaluate`** (returns `{shouldCelebrate, memorizedCount}`, no write) and a separate **`mark…` flip fired from the client mount of the celebration component itself** — so the flag means "has actually been shown," the safety-net stays valid until then, and no new column is needed. General shape: when a boolean guards "user has seen X," write it at the render of X, not at the computation that X is due; otherwise an interruption between decision and display burns the one-shot. (Keep the flip a client-mount Server Action, never an RSC-render side effect — the M5.2b prefetch-flip lesson still binds.)
- **A re-skin's prose brief can supersede the inherited rationale for an existing divergence — record the honest reason, not the stale one (M5.5g).** M5.2b had kept the dictionary row gloss Spanish and filed it under "Spanish-first override / standing rule." The M5.5g brief corrected the *attribution*: the row shows Spanish because the dictionary is an index of already-memorized words (Spanish reinforces; French stays at the fiche), not because of a blanket Spanish-first rule. Same shipped behavior, but the recorded *why* was wrong — and a wrong rationale propagates into future decisions. When a brief re-grounds a decision you'd have justified from memory, update the doc's reasoning to match the real one and say it changed.
- **"Extract one new helper" — grep for it first; it may already exist, and the honest move is consolidate-and-route, not invent-a-second (M5.5h).** The drill-repaint brief asked to extract a single `tenseLabel`, consumed by every surface "so no tense string is inlined." But a `drillTenseLabel` already existed (pure, tested, regression-locked) doing exactly that for four of the surfaces — only the setup chips still read an inline `DRILL_TENSES[].label`. The right shape wasn't a fresh parallel helper; it was to formalize the canonical one (`tenseLabel`), route the last hold-out (the chips) through it, and **keep the pre-existing short `drillTenseLabel` for its one other caller** (`lib/review-cue.ts`'s compact cue chip, which genuinely wants the short form) rather than forcing one helper across two surfaces with different length needs. The tell you're about to fork: you're writing a "new" helper whose job already has an owner — search the codebase for the function before extracting, then decide consolidate vs. coexist *per caller's actual need*, and lock the mapping with a test. (Same reuse family as shared-`posAbbrev` / shared-`WordRow`; the inverse of the fork trap — here the risk was a needless second copy of a helper that already shipped.)
- **A terminology rename for one screen ripples to every surface that shares the label source — confirm scope, then rename at the source (M5.5h).** Renaming the drill's `Pretérito perfecto simple` → `Pretérito indefinido` looked drill-local, but the same tense name is shown twice on the result screen: the drill's own `tenseLabel` AND the **shared** `ConjugationGrid.labelEs` (also rendered in /add + /review verb tables). A drill-only rename would have left the grid header on that same screen reading the old name — self-inconsistent — and the grid is reused, not forked, so "drill-only" wasn't expressible without a fork. Surfaced the two label sources and the blast radius, got Omar's app-wide call, renamed both (display only — `resolveGridTense` still accepts the old annotation as parser *input*), and updated the grid tests. When asked to rename a shared label, enumerate every surface that reads it before editing, and rename the source rather than the call site.
