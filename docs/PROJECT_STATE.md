## What Paco is

A Spanish vocabulary learning app for French speakers (A2-B1 level). User adds Spanish words; the app generates bilingual definitions, contextual examples, confusable distractors, and lemma metadata via Anthropic. Words enter an FSRS spaced-repetition schedule. Daily review sessions mix fill-in-blank and multiple-choice formats with auto-rated answers the user can override.

The product is mobile-first, French UI, designed for the user (Omar) to actually use daily — not a demo or portfolio piece.

- **Spanish-first definitions** (M2.5): definition stored as `{es, fr, pos?}` JSONB. Spanish primary in UI. French behind a "Voir en français" reveal that counts as a hint in MCQ.

### From M5.7 (v0.8.11) — Press / interaction-feedback states (cross-cutting foundation)

A **cross-cutting** interaction-feedback foundation (guidance-driven, `GUIDANCE-press-states.md`; NOT a per-screen cluster). Before this, almost nothing reacted to a tap — only the review exercises gave feedback — so the user couldn't tell what was actionable. M5.7 adds a single consistent press vocabulary on every actionable surface, additive only (no layout, rest-colour, or copy change). Suite 454 (unchanged — CSS + className-only); tsc + lint clean. Device-smoke-tested across all surfaces, all 4 palettes (incl. Nuit), and reduce-motion; tagged v0.8.11 on the close commit.

- **ONE reusable CSS primitive — six `press-*` classes in `globals.css`** (`press-card`, `press-card-feature`, `press-row`, `press-pill`, `press-pill-amber`, `press-icon`), all derived from the EXISTING palette (no new hue). Three layered levers per the spec: (a) a darkening **veil** — an `::after` overlay at `var(--color-ink)` (6% normal / 10% on already-tinted crème+ surfaces); because `--color-ink` flips light on Nuit, the veil reads as a contrast press-highlight on every theme rather than always darkening; (b) a **depress** (`scale(0.985)` cards / `0.94` small targets); (c) a **shadow reduction** on cards. `press-pill-amber` darkens the amber FILL to `--color-amber-deep` instead of overlaying a veil (it has no light surface to darken). Veil press-in is near-immediate (40ms), release eases out (110ms).
- **Pure CSS `:active` / `:focus-visible`, deliberately NOT a JS `Pressable` wrapper.** Reason: a JS press component would force `'use client'` onto cards/rows that are Server Components today (`HubCard`, `WordRow`, `CollectionSection`, the dictionary index). `:active` keeps them server-side and handles pointercancel / scroll-cancel natively (a drag that becomes a scroll releases `:active` for free). The accessibility focus ring is a global `a/button/[role=button]/[tabindex]:focus-visible` rule, themed via `color-mix(in srgb, var(--color-accent) 35%, transparent)`. `prefers-reduced-motion: reduce` keeps the colour veil / fill change but drops the scale/translate.
- **Wired across the app:** `Button` (a depress `active:scale-[0.98]` added on top of its existing §03 colour/shadow/translate press states; a native disabled `<button>` never receives `:active` so it stays inert without a guard); `WordRow` (scoped to the **non-`flush`, linked** row only — the `flush` variant is wrapped by `SwipeRow`, whose pointer-drag owns the gesture, so a press veil there would flash as the swipe starts); `TopNav` (home/avatar round icons → `press-icon`/`press-pill-amber` by active state; all nav pills → `press-pill` / `press-pill-amber`, incl. the locked dict pill which still navigates to the locked screen); Home `HubCard` (`press-card` / `press-card-feature`) + `CollectionSection` header & footer (`press-row`); dictionary entry rows; discover suggestion cards (`press-card`) + topic cards (replaced their ad-hoc `active:bg-tint` with the standard primitive) + close icon; account `ThemePicker` swatches + `components/form` Controls (Stepper, Segmented) + the shared `LinkRow` (account + legal rows); word-detail ⋮ trigger + menu items; `AudioButton` circle.
- **Deliberately untouched:** the **locked hub cards and gated controls get NO press class** (non-actionable — the gate is the message); the **review exercise components** (`RatingButtons`, `FillInBlank`, `MultipleChoice`, the session flow) already had feedback (the one surface the guidance names as working), so they were left alone; the swipe Supprimer panel stays owned by its drag gesture.
- **Two honest gaps, logged to `backlog.md` (Mobile UX polish), not silently dropped.** (a) **Hit-target ≥44px** — the guidance lists it but also forbids layout changes ("ajoute uniquement le feedback d'interaction"); several pills/icons sit at 32–38px, and expanding them is a layout pass, so sizes were left as-is. (b) The `/words` **sort text-toggles** and the ephemeral discover **toast-dismiss ×** were left focus-ring-only — a veil over tiny inline text / a caret reads worse than nothing.
- **Post-device-test tuning (v0.8.12).** The first cut (v0.8.11) shipped the values too quiet to perceive — veil `ink @6%`, `scale .985` — so on a real screen taps looked dead (next to the review buttons' full `bg-tint` swap).
